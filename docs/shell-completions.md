# Shell completions

clawctl provides tab completion for bash and zsh, covering commands, options,
instance names, and `openclaw` subcommands.

## Setup

Add one line to your shell rc file:

```bash
# Bash — ~/.bashrc:
eval "$(clawctl completions bash)"

# Zsh — ~/.zshrc:
eval "$(clawctl completions zsh)"
```

Restart your shell (or `source` the rc file) to activate.

## What gets completed

| Context                   | Completed values                                      |
| ------------------------- | ----------------------------------------------------- |
| `clawctl <TAB>`           | All commands (`create`, `list`, `start`, `oc`, etc.)  |
| `clawctl start <TAB>`     | Instance names from the registry                      |
| `clawctl start --<TAB>`   | Command-specific options (`--instance`, `--help`)     |
| `clawctl delete --<TAB>`  | `--instance`, `--purge`, `--help`                     |
| `clawctl oc <TAB>`        | openclaw subcommands (from cached completion scripts) |
| `clawctl oc config <TAB>` | Deep openclaw subcommand completion (when cached)     |
| `clawctl use <TAB>`       | Instance names                                        |

Instance names are read directly from `~/.config/clawctl/instances.json`
via `python3` (~30ms, no Bun startup cost).

## openclaw subcommand completion

`clawctl oc` (and `clawctl openclaw`) provide full deep completion of
openclaw's own command tree — not just top-level commands, but subcommands,
flags, and arguments. This works by caching openclaw's native completion
scripts from a running VM.

### How the cache works

The completion cache lives at:

```
~/.config/clawctl/oc-completions.zsh
~/.config/clawctl/oc-completions.bash
```

These are the raw completion scripts produced by `openclaw completion --shell
<shell>` inside the VM. clawctl's completion scripts `source` them at load
time, then delegate to openclaw's completion functions when you tab-complete
after `oc`.

### Cache lifecycle

The cache is populated and refreshed **lazily** — never during shell startup,
only when you actually tab-complete `oc` subcommands:

1. **First `oc <TAB>` (no cache)**: The completion script runs `clawctl
completions update-oc` synchronously. This SSHs into the VM, fetches the
   completion scripts, and writes the cache. Takes ~500ms-1s on first use.
   If no VM is running, no completions are shown.

2. **Normal `oc <TAB>` (cache exists, fresh)**: Uses the cached completion
   functions directly. No VM interaction, no delay.

3. **`oc <TAB>` with stale cache (>24 hours)**: Uses the existing cached
   completions immediately, then spawns a background `clawctl completions
update-oc` process to refresh the cache. The update happens silently;
   new completions are picked up on the next `oc <TAB>` without restarting
   the shell.

4. **After a background refresh**: On the next `oc <TAB>`, the completion
   script detects the cache file has a newer mtime than when it was last
   sourced. It re-sources the file inline, making updated completions
   available immediately.

### Manual cache update

```bash
clawctl completions update-oc
```

Fetches completion scripts from the current instance's VM and writes both
bash and zsh caches. Useful after upgrading openclaw in the VM if you don't
want to wait for the 24-hour staleness window.

## How it works internally

The completion scripts are generated by TypeScript templates
(`src/templates/completions/bash.ts` and `zsh.ts`) and printed to stdout by
`clawctl completions <shell>`. The `eval "$(…)"` pattern in the user's rc
file evaluates the output, defining the completion functions in the current
shell session.

Key implementation details:

- **Function name sanitization**: The `binName` (e.g., `clawctl-dev`) has
  hyphens replaced with underscores for shell function names
  (`_clawctl_dev_completions`), since hyphens are invalid in shell identifiers.
  The original name is preserved for `complete -F` / `compdef` registration.

- **Instance name lookup**: Uses `python3` to parse the JSON registry (~30ms)
  rather than spawning `bun`/`clawctl` (~200ms+). `python3` is always
  available on macOS.

- **Cache mtime tracking**: A shell variable (`_clawctl_oc_cache_mtime`)
  records when the cache was last sourced. On each `oc <TAB>`, the completion
  script compares the file's current mtime against this variable to decide
  whether to re-source — enabling hot-reload after background updates without
  restarting the shell.

- **Background refresh**: Uses `&!` (zsh) or `& disown` (bash) to fully
  detach the update process so it doesn't block completion or produce output.

- **No static fallback**: openclaw subcommand completions come exclusively
  from the cached scripts. There is no hardcoded command list to maintain or
  drift out of sync.
